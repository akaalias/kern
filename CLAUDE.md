# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Kern is a macOS text editor — Emacs keybindings/editing model with an iA Writer look. A thin SwiftUI shell launches an editor written in C (SDL2 + OpenGL, `stb_truetype` for fonts). There is **no UI toolkit**: the editor draws immediately through a 12-function renderer interface. (The repo/folder is named `microui` for historical reasons — that dependency was removed; geometry types now come from `Editor/gfx.h`.)

## Development process (non-negotiable)

Every feature and fix follows this exact order — no step is skippable:

1. **Red** — write the failing headless test FIRST (`tests/unit_*.c`, plus any new `platform_stub` controls it needs) and run `make test` to watch it fail.
2. **Green** — implement until the suite passes. Put logic where tests can reach it: pure C behind the seams, even when the caller is Swift (see `kern_feed_skip_post`/`kern_feed_quote_text` — feed logic in C, called from the Swift formatter via the bridging header).
3. **Build** — `xcodebuild` (command in *Commands* below).
4. **Launch** — open the Debug app and confirm a **live pid** (`pgrep -x Kern`).
5. **User verifies** — tell the user what to test and **stop**. No commit, no push, no release until they explicitly confirm it works.
6. **Ship** — commit to `main`, push (the pre-push gate runs), `scripts/release.sh`, commit the pbxproj version bump.

External API details (endpoint paths, params, scopes) are verified against the provider's docs before use — never written from memory.

## Features

**Keep this list current — add an entry whenever a user-facing feature lands.** It exists so a session can answer "do we have X?" without spelunking. Keybindings are tabulated in `README.md`.

- **Daily notes** — `Cmd-Shift-T` jumps to (or creates) today's note at any time; the buffer auto-saves every few seconds while modified — **non-blocking** (`autosave_async`): the buffer is snapshotted on the main thread and the disk write runs on a detached background thread so typing never stalls (one save at a time via `g_saving`; optimistic dirty-clear; synchronous + deterministic under `KERN_HEADLESS_TEST`). **Launch routing** (`editor_main`, `textview.c`): if today's `YYYY-MM-DD.md` doesn't exist yet it's created (seeded with a date heading) and, if a previous session had a file open, a backlink to it is dropped in; if today's note already exists, the last session's file is reopened instead.
- **Session persistence** — the open file and view toggles (typewriter / syntax mask / style mask) are saved to `.kern_session` in the documents dir on each autosave tick and on quit (`session_save`), and restored on launch (`session_load`) — so a relaunch resumes the same file and modes. The startup pin seeds `content_h` + the mono flag before `nav_ensure_cursor_visible` so the restored caret lands at the right strike height. App-only (sandboxed-path-guarded; no-op headless).
- **Per-file cursor memory** — every file remembers its own caret position, persisted to `.kern_positions` (`filepos_*` in `textview.c`; a `{path,line,col}` table). The position is stashed on leaving a file, every autosave tick, and on quit, and restored whenever the file is reopened — hooked into the central `open_or_create_file` so it covers wikilink-follow, buffer switch (`C-x b`), the daily-note jump, and the launch routing.
- **Emacs editing model** — movement, kill/yank ring, operation-based grouped undo (`C-/`), open-line, transpose, word case ops (`editing.c` / `undo.c`, dispatched via `commands.c`). `C-n`/`C-p` and the up/down arrows move by **visual** row (wrapped paragraphs step one screen line at a time), keeping a pixel goal column — `nav_visual_move` in `navigation.c`.
- **Mark & region** — `C-Space` mark, cut/copy/yank synced with the macOS clipboard, select-whole-buffer, exchange point/mark. **Backspace/Delete over an active selection delete the whole region** (without touching the kill ring) — `ed_delete_region` in `editing.c`, called from `cmd_backspace`/`cmd_delete` (falls back to the single-char delete when there's no region).
- **Wrap selection** — with a region marked, typing a markdown emphasis/code/highlight char (`*`, `_`, `` ` ``, `=`) **surrounds** the region with that marker on both ends instead of replacing it, and leaves the region active over the inner text — so typing `*` twice turns a selection into `**bold**`, and `=` twice into `==highlight==`. Core is `ed_wrap_region(ed, open, close)` in `editing.c` (insert close-then-open as one undo group, re-anchor mark/cursor onto the inner text preserving orientation; no-op on an empty/markless region, returning 0 so the caller falls back to a normal insert). Triggered from the `SDL_TEXTINPUT` path in `textview.c` (gated on `mark_active` + a single-byte wrap char). Headless-tested in `tests/unit_editing.c` (`ed_wrap_region` directly) and `tests/unit_textview.c` (the real typed-`**` event flow).
- **Search** — incremental `C-s`/`C-r` with match highlights (`navigation.c`).
- **Live markdown** — headings / bold (`**`) / italic (`_`) / highlight (`==`) / inline code (`` ` ``) / **underline (`++text++`)** styled as you type (`md_render.c`; the span kinds are the `SP_*` enum, underline is `SP_UNDERLINE` — drawn as a solid baseline rule in the text's own color, no width change so caret/click parity is automatic). **`> ` blockquote lines render in italic** (classic quote styling): `md_is_blockquote` gives the line an italic *base* style in both `md_draw_text` and `md_x_to_col` (so click/caret parity holds even when italic measures wider — tested with `stub_set_style_extra`), inline spans still override per glyph (bold stays bold), and the `> ` marker dims like other delimiters. **`==highlight==` is drawn as a yellow over+under rule** (a line along the top edge `y+1` and the bottom edge `y+font_h-1`), not a full-height fill; its bottom rule is one row below the `++` baseline underline (`font_h*0.92`) so a run that is both highlighted and underlined shows two distinct lines rather than one overwriting the other. (The span model is flat — `md_scan_spans` resumes after each span, so a single glyph is only ever one `SP_*` kind; the distinct rows keep the two rules from colliding by design.) The wrap-selection chars include `+` so a marked region + `+`×2 becomes `++underline++`. **`Cmd-Shift-H`** / **`Cmd-Shift-U`** toggle a `==highlight==` / `++underline++` around the **sentence the caret is in** — no selection needed: `ed_toggle_sentence_highlight` / `ed_toggle_sentence_underline` (both thin wrappers over `ed_toggle_sentence_wrap(ed, '='|'+')` in `editing.c`) find the sentence bounds on the caret's logical line (previous `.!?` → next `.!?`, with surrounding spaces/markers trimmed **both** ends so a no-terminator sentence toggles off cleanly), and either wrap it or, if already wrapped in that marker, remove it — one undo group, no-op in the read-only Context section. Detection looks at the markers immediately hugging the sentence, so highlight and underline nest and each toggles independently. Headless-tested in `tests/unit_editing.c` (wrap/unwrap/undo/neighbors/no-terminator/underline+nesting) and the key routing in `tests/unit_textview.c`.
- **Notes & wikilinks** — `[[` autocomplete, `Cmd-Enter` follows the link, `Cmd-Shift-←/→` history; `Cmd-Shift-N` extracts the selection into a new linked note (`textview.c`, `recent.c`). The `[[` autocomplete (`buf_list_matches` in `buffer.c`, capped at `WL_MAX`=5) matches **note files only** (`.md`/`.markdown`/`.txt`), **excludes the current file** (MRU index 0), and works two ways: a **bare `[[`** lists the most-recently-opened notes (MRU order, recent doc on top — `recency_rank` consults `recent.c`); **typing** filters to notes whose name *contains* the query as a case-insensitive **substring** (`match_score`/`find_ci` — "Hello" → "Hello World"/"My Hello", deliberately NOT a loose subsequence that would match "histories"), ranked prefix > word-start > mid-word, then recency, then alphabetically. (`buffer.c` links `recent.c` — added to the perf harness too.) **Link resolution is extension-insensitive, like Obsidian** (`buf_resolve_note_path` in `buffer.c`, used by `open_or_create_file` — so it covers Cmd-Enter follow, find-file, buffer switch, graph clicks): `[[My File]]` and `[[My File.md]]` open the same note — a name ending in a note extension or existing verbatim is used as-is, else an existing sibling with `.md`/`.markdown`/`.txt` appended wins, else the note is **created as `.md`** (so a followed ghost link becomes a real note visible to the autocomplete/graph, not an extensionless orphan). Headless-tested in `tests/unit_buffer.c` + `tests/unit_textview.c`.
- **Context section** — an auto-generated, **read-only** "Context" block appended below every document on open, listing related notes: **backlinks** (other notes whose text contains a `[[ThisNote]]` wikilink), notes **created the same calendar day** (by file birth time), and **"Opened after"** — the notes that were open right before this one was opened (see the *Opened-after history* entry below). A **dotted divider with a centered "Context" label** (drawn in `do_render` at `readonly_from + 1`, with blank spacing lines above/below) separates it from the editable page — there's no literal `## Context` heading line. The **caret turns warm amber** (the complement of its default cyan) whenever it's inside the section, signalling read-only. `M->`/`Cmd-S-.` (bottom of document) lands the caret at the **true end** — the last line of the Context section (read-only but navigable); **select-whole-buffer** (`C-x h`) still stops at the content boundary (`buf_content_line_count`) so a select-all + cut can't touch the section. Real buffer lines, but static: `EditorState.readonly_from` marks where the section begins, the `ed_*` editing primitives refuse to mutate any line `>= readonly_from` (so the caret can still move in and follow wikilinks, but nothing's editable), and save/serialize stop at `buf_content_line_count()` so the section never hits disk or the X-publish payload. The boundary is kept in sync as content above it changes by `buf_insert_line_at`/`buf_delete_line_at`. Generated in `context_refresh()` (`textview.c`, app-only — scans the documents dir + creation times; no-op under `KERN_HEADLESS_TEST`, where the read-only/save machinery is unit-tested directly via `readonly_from`). Margin-note footnote definitions insert at `readonly_from` (above the section) so they stay editable and saved.
- **Opened-after history** — every note remembers the files that were open *right before* it was opened, surfaced as the **"Opened after"** list in the Context section. The transition is recorded whenever the open file changes — `C-x C-f` find-file, `C-x C-b` buffer switch, a `Cmd-Enter` wikilink follow (and back/forward history), and the `Cmd-Shift-T` daily-note jump all funnel through `open_or_create_file`/`cmd_open_daily_note`, which call `opened_after_record(new, prev)` before `context_refresh()`. Kept as a per-file table of predecessor **basenames** (`g_opened_after` in `textview.c`, MRU-ordered, deduped, capped at `OA_PRED_MAX`=24), so a repeat open lifts the predecessor to the front rather than duplicating it. Persisted to `.kern_opened_after` in the documents dir (`opened_after_save`/`opened_after_load`, app-only like `.kern_positions`; tab-separated `predbase\tfullpath`, oldest-first so a reload rebuilds the same MRU order). `opened_after_list` lists the predecessors verbatim (**no on-disk existence filter** — a note opened via `C-x C-f` may not be saved to disk yet, e.g. a blank new note that lives only in memory until autosave, but it was genuinely opened so it belongs in the list; a `[[wikilink]]` to a not-yet-existing note just creates it on follow, as everywhere). The record/table logic compiles into both builds and is headless-tested in `tests/unit_textview.c` (the `tv_test_opened_after` seam) — predecessor recording, most-recent-first order, and dedup-by-lift — while the rendering is app-only.
- **Graph view** — `C-x g` (View → Graph View) opens a modal, Obsidian-style force-directed map of the vault's relationships over a blurred backdrop: one node per note (ghost nodes for wikilink targets not on disk yet render as a **gray outline ring** — an AA `r_draw_line` polyline, hairline at any radius — instead of a filled disc, so not-yet-created notes read at a glance; the open note in the read-only amber, hover in caret cyan; **radius grows with the note's backlink count** — direction-aware references TO the note: incoming wikilinks (deduped per source file, credited by `graph_scan_links`) plus the context relations via `graph_add_backlink` (same-day companions, appearing in another note's opened-after list) — so much-referenced notes read as hubs), edges typed by relation — **solid** = `[[wikilink]]`, **dashed** = opened-after, **dotted** = same creation day (exactly the Context section's relations; a legend sits bottom-left). **Click a node to open that note** (same save + history push as a wikilink follow), **drag a node** to rearrange (live physics), **drag empty background to pan** the view (takes the view over like the wheel, so auto-fit stops), **scroll** to zoom (0.2–4×), Esc or `C-x g` closes. The overlay **opens fitted** — the settled layout is centered and scaled so every node is on screen (`graph_bounds` + `graph_fit_view`; refit continues while the sim moves until the user wheels, which hands the zoom over) — and **labels and edges fade in with zoom** (labels hidden ≤0.8× / full ≥1.2×; edges on a lower ramp ≤0.55× / ≥1.0× with a **faint floor** — 25% of base alpha — so the fitted overview still shows the structure; the hovered and current nodes' labels — and a hovered node's edges — always show in full). Model + deterministic layout live in `Editor/graph.c`/`graph.h` (pure C: extension-insensitive node dedup, kind-bitmask edge dedup, `[[link]]` scanner, golden-angle seed (degree-1 nodes seeded beside their only neighbor, radially outward) + damped softened-inverse-square-repulsion/linear-spring/center-gravity embedder — springs are degree-normalized per endpoint (a many-edged hub has bounded stiffness, else it vcap-oscillates and never sleeps), degree-1 satellites pull double and repel half (so single-connection notes settle beside their one neighbor instead of stranded across the map), and sub-0.75px/step velocities snap to zero (micro-limit-cycles would keep the sim awake at 60fps forever); **no rand()**, so layouts reproduce exactly; in `CORE`+`COVCORE`, suite `tests/unit_graph.c`). The overlay half is `graph_*` in `textview.c`: `graph_build()` scans the docs dir (compiled in both builds; creation day falls back to mtime off-Darwin) + the `g_opened_after` table, `cmd_toggle_graph` pre-settles 150 steps so it opens laid out, the tick keeps relaxing while `graph_energy > GRAPH_SETTLE` (drag/resize re-excite; ORed into the `SDL_WaitEventTimeout` wake), and `draw_graph_overlay` renders with two renderer-seam additions built for hundreds-per-frame use: `r_draw_line` (anti-aliased at any angle — a solid core quad + 1px alpha-feathered strips via per-vertex colors, no texture uploads; dashes are short lines, dots are small circles) and `r_draw_circle` (one cached mipmapped 64×64 alpha-disc texture, scaled per call — never rebuild-per-call like `r_draw_round_rect`), labels at `r_set_font_scale(0.6)`; the headless stubs record both as bounding rects. Headless-tested in `tests/unit_textview.c` (open/toggle/modal-swallow, click-to-open via the `tv_test_graph_node_screen` seam, DAY + OPENED edges, menu bridge).
- **Files** — find/open with inline completion, save, save-as, recent-buffer switch (MRU); all paths sandboxed under the app container.
- **View** — font size `⌘=`/`⌘-`, recenter `C-l`, goto-line `M-g`, list indent/outdent (`Tab`/`Shift-Tab`).
- **Typewriter mode** — `C-x t` toggles; the active line pins at the golden ratio (~38% down the page) and the buffer scrolls under the cursor. Virtual whitespace below EOF lets the last line still pin. Shares `nav_pin_cursor` with `C-l` recenter; flag is `ViewState.typewriter_mode` (`navigation.c` / `commands.c`). The caret is **also pinned horizontally** at the page-column center: the whole text pane is translated by `-ViewState.scroll_x` (eased with `HSCROLL_EASE`) so the caret holds a fixed screen x while the page slides under it; geometry stays in natural coords — only the render/highlight origins subtract `scroll_x` and `nav_click_to_cursor` adds it back. A **hard right margin** blocks typing past the writable width (`nav_at_right_margin`) so lines don't auto-wrap onto a new row — only Enter starts a new line. The page is dressed with guides (dotted writing-area edges + solid page margins, tracking `scroll_x`) — the same fixed framed page as normal mode (see **Page furniture**; a closed rectangle with top/bottom edges) — and two **frosted-plastic guards** flanking the strike point plus a **third panel** covering everything below them to the content bottom (`draw_typewriter_fog` in `textview.c`): each region real-gaussian-blurs the text under it (`r_blur_rect` — framebuffer copy + weighted-additive kernel, scissor-clipped) plus a tint; the two guards are additionally stencil-masked to a rounded shape (`r_clip_mask_*`) and pinned at the fixed strike line so they don't bounce as the page glides. `r_blur_rect`/`r_clip_mask_*` are real-GL additions below the renderer seam (app-only; headless stubs in `tests/stub_renderer.c`). **Virtual-space carriage:** the caret can sit *past* end-of-text, holding the horizontal strike point over blank page like a real carriage (`ViewState.virtual_col` = columns past EOL; live only while the caret is still where the move that set it left it — `tv_virtual_cols`). A vertical move (`nav_visual_move`) onto a shorter line floats the caret to the goal column instead of snapping to EOL; the horizontal pin is the caret's real x **plus** the float (`tv_virtual_px`), so vertical moves hold the page's horizontal place and arrow floats glide it. **Left/right arrows are horizontal-only, up/down vertical-only** (`tv_horizontal_move`): right past EOL floats (capped at the right margin), left retracts then walks the text, neither jumps lines (shift-select / word-jump keep normal text movement). Typing while floated pads the gap with spaces as one undoable insert. Typewriter mode also **hides the scrollbar** and renders **everything in mono** (`iAWriterMonoS`): `md_set_force_mono` makes `md_draw_text`/`md_x_to_col` draw every glyph in `FONT_MONO`, and `md_wrap_font_style()` (FONT_MONO when forced, else FONT_REGULAR) is what `navigation.c`'s `body_width`/`nav_get_wrap_breaks`/`nav_at_right_margin` measure with, so wrap/click/render stay in lockstep; the `C-x t` toggle flips the flag and invalidates all wraps to reflow in the new font.
- **Syntax highlighting** — `C-x y` toggles part-of-speech highlighting. Unlike iA Writer's five-hue rainbow, the palette is **value, not hue** (Tufte): a luminance ramp by grammatical importance — verb (brightest) > noun > adjective > adverb >> function words (determiner/preposition/pronoun/conjunction/particle, dimmed so the grammatical glue recedes). What you read is the prose's content rhythm, figure vs ground. Tagging is behind a seam (`pos_tagger.h`): the app backs it with `NSLinguisticTagger` lexical-class scheme (`pos_tagger_nl.m` — same engine iA uses), tests with a fixture lexicon (`tests/pos_tagger_fake.c`). Per-line spans are lazily cached on the `Line` (`pos_spans`, invalidated by `line_dirty` like `md_spans`); `pos_render.c` owns the cache, the value palette, and a determiner→noun post-correction (a gerund right after "a/the" — "a heading" — is retagged from verb to noun). `md_draw_text` applies the per-span value, layering over markdown so bold/italic keep their weight. Bit-per-class mask is `ViewState.syntax_mask` (0 = off). **Highlight is an isolate, not a hide:** when the mask is non-zero, a shown class takes its ramp value and *everything else* — toggled-off classes and untagged/`POS_OTHER` text — drops to a muted ground (`pos_mute_color()`, one step below the function-word value) so the shown classes pop. The mute is applied in `md_draw_text` (the `else` of the `pos_color_at` branch), not in `pos_color_at` (whose `1=shown / 0=not` contract is unchanged). Per-type toggles live in the **View** menu: Verbs / Nouns / Adjectives / Adverbs / **Function Words** (one item for all five function-word classes, since they share the palette value), each flipping one bit of `syntax_mask` via a `kern_toggle_*` bridge; the master "Syntax Highlighting" (and `C-x y`) flips all bits. **Type-in-progress hold + fade-in:** the word currently being typed would re-tag (and so re-color) letter-by-letter as it grows ("comp"→"compute"→"computer"), which flickers. So the word under the caret is held at the base ("max") color — neither colored nor muted — until it's finished, then fades from the base color up to its real POS color over `POS_FADE_MS` (5s, smoothstep — "ink drying"). A word is "in progress" only because an **edit** put the caret in it (`buf_edit_seq` change in `pos_anim_track`, `textview.c`); it finishes when the caret leaves it — a space, punctuation, an arrow, a click — so merely *navigating* onto already-colored text never disturbs its color. The timing/animation state lives in `pos_render.c` (`pos_set_wip` / `pos_fade_begin` / `pos_resolve_color`, a small fade ring keyed by `(Line*, byte range, t0)`) **not on the `Line`**, since `line_dirty` rebuilds the per-line span cache on every keystroke; `md_draw_text` calls `pos_resolve_color` in place of the raw `pos_color_at`/`pos_mute_color` choice. The fade drives the event loop at ~60fps via a `g_pos_animating` flag (ORed into `SDL_WaitEventTimeout`, beside the scroll/focus flags). Pointer-compared, never dereferenced (stale-pointer-safe, like `sub_render`'s reveal range). Headless-tested in `tests/unit_pos.c` (`pos_word_bounds`, the in-progress hold, the timed lerp).
- **Style check** — `C-x s` toggles iA-Writer-style strike-through of cuttable text. A third per-line span layer (`style_check.c` → `Line.style_spans`, invalidated by `line_dirty` like `md_spans`/`pos_spans`). Each category gets a distinct line texture (`StyleDecor`), drawn in `md_draw_text` as dashes with the `wave` wobble (one `r_draw_rect` per dash): **filler** → greyed strike through the x-height ("delete"); **redundancy** → a wavy underline on the baseline, on only its cuttable word(s) (`cut_from`/`cut_len` per pattern, so "a̶d̶d̶e̶d̶ bonus" keeps "bonus"); **cliché** → a straight dotted underline ("rewrite", text kept readable). Pure C — curated word/phrase lists (fillers, redundancies, clichés/generalizations) in `k_patterns[]`, whole-word/longest-match, no tagger or renderer dependency, fully headless-tested (`tests/unit_style.c`). Categories are `StyleCategory` (`STYLE_FILLER`/`STYLE_REDUNDANCY`/`STYLE_CLICHE`); bit-per-category mask is `ViewState.style_mask` (0 = off). Detection is precision-first (context-free entries only).
- **Symbol substitution** — `C-x l` toggles display-only text→symbol rendering ("ligatures"): the buffer keeps the literal bytes but the renderer draws a single glyph in their place — `->`→→, `<-`/`=>`/`<=>`, `!=`/`<=`/`>=`/`~=`, `--`→— (em), a digit-`-`-digit en dash, `...`→…, `+-`→±, `(c)`/`(r)`/`(tm)`, directional smart quotes `"`/`'` (the close form also gives the right apostrophe in "don't"), **Greek by name** (`lambda`→λ, `Sigma`→Σ, … — whole-word, case-sensitive), and a few **math operator words** (`forall`→∀, `exists`→∃, `nabla`→∇, `infinity`→∞, `sqrt`→√). Greek/math are `word` entries in `k_fixed[]`: matched only as whole words (bounded by non-`[A-Za-z0-9_]`) so `lambdas`/`lambda_x` stay literal. English-colliding triggers (`in`, `sum`, `and`, `or`, `partial`, the short Greek `eta`/`nu`/`xi`/`chi`) are **deliberately excluded** — no clean prose-vs-math disambiguation, and reveal-on-contact can't rescue a false positive you didn't notice. **Font-glyph fallback:** the bundled body font has partial Unicode coverage (it lacks e.g. ⇒ ⇔ ∀ ∃ ∇ Ω), so `sub_render`'s `emit` skips any substitution whose glyph `r_has_glyph` (a renderer-seam call → `stbtt_FindGlyphIndex`) reports the font can't draw — the source renders as literal text instead of a tofu box. This makes `sub_render.c` renderer-dependent (was pure C); the test stub's `r_has_glyph` returns present-unless-`stub_set_glyph_missing`, so the table is still exercised headlessly. A **fourth** per-line span layer (`sub_render.c` → `Line.sub_spans`, invalidated by `line_dirty` like the other three) — `SubSpan{start,len,category,glyph}`, scanned longest-match-first from a curated table (`k_fixed[]`) plus a little context logic for quotes/en-dash. **Reveal-on-contact**: a token is drawn collapsed (the glyph) unless the caret touches it `[start, start+len]`, in which case the render/measure walks draw the literal source bytes so it stays editable. The caret's line+column is published each frame via `sub_set_caret` (and at event time by the click/vertical-move paths); `sub_reveal_col(l)` returns the caret column for the caret's line (compared by pointer, never dereferenced — a stale value after a line-array realloc is harmless), -1 elsewhere. The geometry-critical invariant is that the two render/measure walks — `md_draw_text` (render + caret-x) and `md_x_to_col` (click + vertical move) — agree on every token's width *and* its reveal state; both read `sub_line_spans` + the active mask (`sub_active_mask`, set per frame from `ViewState.sub_mask`) + `sub_reveal_col`. **Wrap (`nav_get_wrap_breaks`) deliberately measures the literal text, ignoring substitution** — a collapsed glyph is never wider than its source, so a literal-width wrap point always fits when drawn collapsed, and revealing a token redraws exactly the width wrap already reserved → no reflow as the caret moves, and the wrap cache stays valid with no toggle-time invalidation. Pure C, fully headless-tested incl. draw↔click parity, reveal-on-contact, selection-range reveal, and wrap substitution-independence (`tests/unit_sub.c`). **Reveal is a range, not a point:** the active selection's extent on a line (unioned with the caret point) reveals every token it overlaps, so a selected symbol stays expanded instead of flip-flopping as the caret passes — `nav_sub_reveal_for_line(ed, ln)` computes it (caret ∪ `buf_region_ordered` extent) and the render/measure passes call it **per visual row**. Two ordering rules keep the marked region from looking wonky around symbols: (1) `do_render` sets `sub_set_mask` *before* `process_frame`, so the selection/search highlights (`md_col_x`) measure the same collapsed/revealed widths the text pass draws; (2) every `md_draw_text`/`md_col_x`/`md_x_to_col` site sets the per-line reveal first. Categories are `SubCategory` (`SUB_PUNCT`/`SUB_ARROW`/`SUB_GREEK`/`SUB_MATH`/**`SUB_LIGATURE`**); mask is `ViewState.sub_mask`, defaulted to `SUB_MASK_ALL` in `editor_main` (**on by default**; `tv_test_reset` leaves it 0 so the suite's other geometry tests are unaffected, and `unit_sub.c` sets the mask explicitly). Not yet in the View menu (keyboard-only for now). **Typographic f-ligatures** (`ff` `fi` `fl` `ffi` `ffl` → ﬀ ﬁ ﬂ ﬃ ﬄ) are `SUB_LIGATURE` entries in `k_fixed[]` — **not** whole-word (they sit mid-word in "offer"/"file"), longest-match-first so ffi/ffl beat ff. They render only when the bundled font supplies the Alphabetic-Presentation-Forms glyph (else the `r_has_glyph` fallback leaves the source literal).
- **Page furniture** — the page borders (dotted writing-area edges, two-line note gutters, solid page-margin rails, top/bottom edges) and **margin notes** render in **both** typewriter and normal (non-typewriter) editing mode; in normal mode this is gated on the window being wide enough to hold the note strip (`nav_page_margin() >= tv_margin_pad()`), otherwise notes fall back to bottom-of-document footnotes. The borders live in `draw_page_furniture` (split out of the typewriter fog). **One drawing for all modes:** a fixed **framed page** — a closed rectangle filling the writing area (top of content down to just above the status bar), with the vertical rails clipped to the frame and solid **top/bottom edges** closing it off (no scroll-tracking paper edges, no full-window rails). Normal mode gets a top page margin via a negative `min_scroll` (reusing typewriter's whitespace-above mechanism — no nav geometry changes) and a bottom margin by trimming `content_h`. Default font size is 24pt.
- **Page-border toggle** — **`C-x p`** (`page_furniture_hidden` in `ViewState`) hides/shows *all* page furniture — rails, dotted edges, and note gutters — in **both** normal and typewriter mode (gates the `draw_page_furniture()` calls in `do_render`; margin notes and the typewriter frost still draw). Persisted across launches in `.kern_session` (the `noborders` line, alongside `typewriter`/`syntax`/`style`). Key routing is in `handle_cx_prefix_key`; headless-tested in `tests/unit_textview.c` (`test_cx_p_toggles_page_furniture`).
- **Margin notes / footnotes** — `Cmd-Shift-M` (no selection needed; typewriter mode or a wide-enough normal-mode window) opens a keyboard-first margin-note input: the hitzone slides to the right page margin (document frosts behind it, single caret), you type, and **Enter** inserts a `[^id]` footnote marker at the write-head and appends a `[^id]: note` definition at the document bottom (**Esc** cancels). Stored as a **standard markdown footnote**, so in normal mode it's just a footnote line at the bottom; in typewriter mode `draw_margin_notes` (`textview.c`) renders each definition in the right-margin strip aligned to its reference's *visual* row (mono, **80% of the body font size and line-height** (`MN_SCALE`), word-wrapped — via `r_set_font_scale`, a quad-level draw/measure scale on the renderer seam that reuses the existing atlas, so unlike `r_set_font_size` it's safe per frame). State is `mn_*` file-statics; id is base36 of the clock+seq. **Overlap handling:** notes flow down the right margin; when one would collide with the note above it, it **spills to the left margin** instead. The right/left side is cached per id (`g_fn_sides`) and only recomputed (in scroll-independent document space, `recompute_footnote_sides`) when a footnote id appears that isn't cached yet — i.e. when a new note is written or a new doc loads — so scrolling never reshuffles sides. While a new note is being typed, its live input reserves a growing extent and saved notes within it spill left early. **Two render passes:** saved notes draw *before* `draw_typewriter_fog` (so the frost blurs them with the page), the live input draws *after* (sharp, in the hitzone moved to the margin). Notes live in a padded, two-dotted-line **gutter** in each page margin (`tv_note_col`/`tv_note_gutter`, strip width `tv_margin_pad`); `Cmd-Shift-M` scrolls the page so the right note column lands in the fixed centre hitzone (the page comes to the note). A note is drawn whenever **any part** of it is on screen — `draw_margin_notes` walks the buffer tracking the running visual-row index to get each reference's y even when its marker row is above the viewport — so notes don't vanish early on scroll. **Gotcha:** the margin renderer must never call `r_set_font_size` (a size change rebuilds all four font atlases — doing it per frame lags the whole app); the smaller size comes from `r_set_font_scale`, applied **only inside `draw_note_wrapped`** (set `MN_SCALE` → `draw_wrapped` → restore 1.0). Never leave the scale set while nav geometry runs — `nav_line_height`/`r_get_text_height` and `nav_get_wrap_breaks` are scale-sensitive, so an ambient scale shrinks line heights (mis-anchoring `refY`) and corrupts the body wrap caches. The live-input caret matches the body caret's shape (3px × `MN_SCALE`-scaled text-height). Headless-tested in `tests/unit_textview.c` (the `Cmd-Shift-M` authoring flow: marker + appended definition, Esc-cancel, no-room no-op on a narrow window, and normal-mode authoring on a wide window via `stub_set_metrics`).
- **Dev-build badge** — Debug builds show `Dev v<version>` in the status bar's bottom-right (amber); compiled out of Release via `#ifdef DEBUG` (defined only in the Debug config), so the shipped `/Applications` app shows nothing. Version comes from `kern_app_version()` (`macos_style.m`, reads `CFBundleShortVersionString`).
- **Menu bar** — the **full command set as standard macOS menus**: File (open/save/save-as/switch-buffer/daily note), Edit (undo, cut/copy/paste, select-all, the emacs kill/word/case ops, search), Format (the wrap markers, sentence highlight/underline, list indent), View (typewriter / symbols / page borders / syntax / style toggles + font size, recenter, paging), Go (document ends, goto-line, follow link, back/forward), Notes (extract, margin note, publish, the X downloads), Window ("Open Documents Folder in Finder" → `kern_open_documents_folder()` in `macos_style.m`). Decoration is **NOT an `NSMenuDelegate`** (tried — SwiftUI re-grabs menu delegates, so `menuNeedsUpdate:` fired once and froze every checkmark): `kern_menus_sync` (`Platform/macos_style.m`, driven by the `KernItemSpec` tables) is **polled from `editor_tick`** and sets each toggle's live ✓ from its `kern_*_enabled()` query, appends the chord hint once per item, and strips the AppKit-injected "Enter Full Screen" (it can't drive the SDL window). Items are matched **by title via `kern_find_item`**, which also matches the decorated-title prefix — **gotcha: setting `attributedTitle` makes AppKit rewrite the plain `title` to the full decorated string**, so a bare `itemWithTitle:` stops matching after the first pass (this froze every decorated toggle at its first-tick state). Keep the Swift button titles and the ObjC spec tables in sync ("Cliches" stays ASCII to dodge a Unicode-normalization mismatch). **First-open race:** SwiftUI populates each menu's real items lazily on its first display — mid-open, while menu tracking has the tick frozen — so a timer registered **only in `NSEventTrackingRunLoopMode`** (zero cost outside tracking) re-runs `kern_menus_sync` every 0.1s while a menu is open, re-decorating the swapped-in items live. Menu selection runs synchronously on the main thread during tracking, so actions fire despite the parked loop. Bridges are headless-tested in `tests/unit_textview.c` (per-menu action tests + the suppress-next-text guard).
- **Settings window** — SwiftUI `Settings` scene (⌘,) with two tabs: **X (Twitter)** (the publish account) and **Keyboard Shortcuts**. Each `.tabItem` is a `Label(_, systemImage:)` (paperplane / keyboard) — the SF Symbol is what makes macOS render the **native preferences toolbar tab bar**; a text-only `tabItem` falls back to an ugly segmented control. The shortcuts tab is **pure SwiftUI** (`ShortcutsView` in `App/KernApp.swift`): a plain two-pane layout — a left column of group "tabs" (`Button`s, single-click selection, highlighted when active) and a right detail pane listing that group's bindings as key-caps. The chord specs live in the `shortcutGroups` data table; `ChordView` parses an emacs spec (`C-x C-s`, `M-w / Cmd-C`) into `KeyCap`s (modifier prefixes → ⌃/⌥/⇧/⌘). **Deliberately NOT a `NavigationSplitView`** — that renders an app-sidebar (collapse button, toolbar title) and has a first-click-focus bug; the manual `HStack` avoids both. Fixed `.frame(760×660)`. This whole feature **replaced the old AppKit floating shortcuts panel** (and `kern_make_shortcuts_view` / the `KernKeycap` bento builder, all removed from `macos_style.m`) and its title-bar `info.circle` button — the only title-bar button left is the X publish paper-plane.
- **X (Twitter) publishing** — a labelled **"Publish" title-bar button** (icon + text, built in `Platform/macos_style.m`) publishes the current note (or marked region); it's shown only while an X account is connected (the editor loop polls `kern_x_is_connected()` and calls `kern_titlebar_set_x_connected()` on change). Clicking it calls the `kern_publish_to_x()` bridge in `textview.c`, which opens a **confirmation overlay** rather than posting immediately. Results report via both a macOS notification (`UNUserNotificationCenter`, authorized at launch in `AppDelegate`) and the C status bar. OAuth 2.0 PKCE via a loopback listener, Keychain tokens, **Settings → X** tab to connect. Client id injected at build from gitignored `Config/Secrets.xcconfig` → `Info.plist`. Lives in `App/KernApp.swift` + the `kern_x_*` bridge in `textview.c`; setup in `README.md`. (See the main-thread gotcha below.)
- **X news feed** — `C-x n` downloads the connected account's home timeline into a time-stamped `Twitter-Home-Newsfeed-%Y-%m-%d-%H%M.md` note (matching `# Twitter Home Newsfeed — …` title heading) in the documents dir and opens it. One `## <name> — <date> at <HH:MM>` entry per post, followed by the full text as a `> ` **blockquote** (every line prefixed via `kern_feed_quote_text` in `textview.c` — pure C, headless-tested, called from Swift through the bridging header) and the post URL (the byline line was dropped 2026-07-06 — the URL is the only trailer now) — so a post can be marked up and extracted with `Cmd-Shift-N` like any other text. **Noise filter:** posts whose text is only link(s) (bare image posts — their whole text is the t.co media URL) and single-line posts are skipped — `kern_feed_skip_post` in `textview.c` (pure C, headless-tested in `tests/unit_textview.c`; a trailing `\n` is not a second line, and a post whose every token is a URL has no content), called per post by the Swift formatter through the bridging header. Fetch + markdown formatting live in Swift (`XAuth.homeTimeline` in `App/KernApp.swift`: `GET /2/users/{id}/timelines/reverse_chronological?expansions=author_id`, entities unescaped; the numeric user id is resolved lazily from `/2/users/me` and persisted in `XTokens.userID` for accounts connected before the feature). The async result crosses to C via `kern_x_feed_done(ok, markdown)` (any thread; stashed in `feed_result`/`feed_result_text` and applied by `feed_apply_result()` on the main-thread tick, which writes the file via `buf_save_text` + `open_or_create_file`s it — same pattern as the publish result). Key routing is in `handle_cx_prefix_key` (`cmd_fetch_news`, guards on `kern_x_is_connected`). Headless-tested in `tests/unit_textview.c` (not-connected guard, fetch request, note creation + disk write, error reporting) via the `kern_x_fetch_feed` stub in `tests/platform_stub.c`. Note: reading the timeline requires an X API tier with read access (the free write-only tier 429/403s; the error surfaces in the status bar).
- **X bookmarks** — `C-x m` ("marks") downloads **all** of the account's bookmarks into a time-stamped `Twitter-Bookmarks-%Y-%m-%d-%H%M.md` note (matching `# Twitter Bookmarks — …` heading), same entry format as the news feed but **unfiltered** (bookmarks were marked deliberately, so image-only posts and one-liners stay). Swift side (`XAuth.bookmarks` in `App/KernApp.swift`): `GET /2/users/{id}/bookmarks?max_results=50`, paginated via `pagination_token`/`meta.next_token` until exhausted (80-page safety cap; **50, not 100 — at `max_results=100` this endpoint silently truncates and omits `next_token`**, so a pull stops after one page), per-page progress via the status bar; a mid-pagination failure returns the pages already fetched with a trailing "stopped early" note instead of throwing everything away. Result crosses to C via `kern_x_bookmarks_done` into the news feed's kind-aware result plumbing (`feed_done`/`feed_result_kind` in `textview.c` — one apply path, the kind picks filename/title/status). **Requires the `bookmark.read` OAuth scope** (added to `XAuth.scopes`): accounts connected before this feature must disconnect + reconnect in Settings → X, or the API rejects the call. Bookmark endpoints need a Basic-or-higher API tier, like timeline reads. Headless-tested in `tests/unit_textview.c` (guard, request, note creation) via the `kern_x_fetch_bookmarks` stub.
- **Publish-as-reply** — a note that contains an X status URL **followed by commentary** (the workflow: grab a feed entry with `Cmd-Shift-N`, write your take under the tweet's URL) publishes the commentary as a **reply to that tweet** instead of a fresh post. Detection is `kern_reply_scan` in `textview.c` (pure C, headless-tested): the **last** `x.com|twitter.com/<handle>/status/<digits>` URL in the text with non-blank text after it wins; the trailing text (whitespace-trimmed) becomes the tweet body, the id + author handle are stashed (`pub_reply_id`/`pub_reply_handle`). The scan also reconstructs the **original post** from the feed-entry structure above the URL (the `## <name> — <date>` heading → author + date, prettified to "Jul 6" when it parses; the `> ` blockquote → the quoted text, prefixes stripped) into a `KernReplyTarget` struct (`editor_loop.h`). The confirmation overlay then simulates the whole exchange, X-composer style: the "Replying to @handle" caption (dim lead-in, X-blue handle), the **quoted original post** (initials disc — or your real avatar when replying to yourself — bold author, `@handle · date`, text clipped to 200px) with a **thread-connector line** down to your own avatar row, then **only your commentary** as the reply body. A hand-written URL without the feed structure still replies, just without the quoted preview. **The preview is then verified against the API:** opening a reply overlay also fires `kern_x_fetch_tweet(id)` (Swift `XAuth.fetchTweet`: `GET /2/tweets/:id?expansions=author_id&tweet.fields=created_at,note_tweet&user.fields=name,username,profile_image_url` — `note_tweet.text` is the full text for long posts; scopes `tweet.read`+`users.read`, already granted); the result crosses back via `kern_x_tweet_done(ok, id, name, handle, date, text)` (bridging header; any thread, stashed, applied by `pub_apply_tweet()` on the tick), replacing the note-parsed author/handle/date/quote with the API truth and downloading the author's real avatar (`kern_x_reply_avatar_rgba`, same C-owned-RGBA scheme as the account photo, cleared per lookup). A failed lookup (e.g. write-only API tier) or a stale id (re-publish raced it, overlay closed) is dropped — the note-parsed preview stays. Confirm passes the id through the bridge — `kern_x_publish(text, in_reply_to)` (NULL = plain post) → Swift `post(text:replyTo:)` adds `"reply": {"in_reply_to_tweet_id": id}` to the `POST /2/tweets` body (verified against docs.x.com; needs only the already-granted `tweet.read`/`tweet.write` scopes). No URL, or nothing after it → the whole note posts exactly as before. Works on a marked region too (the scan runs on whatever was snapshotted). Headless-tested in `tests/unit_textview.c` (scan extraction/last-URL-wins/negatives, reply-mode overlay state, the id reaching the publish stub, plain-post regression).
- **Publish-as-quote** — the mirror of publish-as-reply: plain commentary **above** a single feed entry (the `## <name> — <date>` heading + `> ` blockquote + status URL, with **nothing after the URL**) publishes as a **quote post**. `kern_quote_scan` in `textview.c` (pure C, headless-tested; declared beside `kern_reply_scan` in `editor_loop.h`) matches that shape and declines when text follows the URL (reply wins — the more deliberate act), when nothing precedes the entry, or when the commentary itself contains another entry (a second `/status/` URL or a `>`-prefixed line — multi-entry feed notes are ambiguous). Posting **appends the tweet URL to the text** (`kern_x_publish(text+"\n\n"+url, NULL)`) — X renders a trailing permalink as the quote card; the API's `quote_tweet_id` field is **deliberately not used** (Enterprise-gated / blocked cross-account on self-serve since early 2026). The overlay flips the arrangement: your identity row + commentary on top, the original post below in a **hairline-bordered rounded inner card** (28px avatar inline with the author row, clipped text) — `pub_kind` (`PUBK_POST/REPLY/QUOTE`, seam `tv_test_pub_kind`) picks the geometry in `pub_layout`, and `draw_quote_post` renders the original for both modes (reply: full-size block + thread connector above; quote: the inner card below). The API tweet-lookup (`kern_x_fetch_tweet`) fires for quotes too, so the card shows the authoritative content + real avatar.
- **Error log note** — async X failures (publish, feed/bookmarks download) are appended to a **`Kern-Errors.md` note in the documents dir** — a `## <context> — <date time>` heading + the full error as a `> ` blockquote (via `kern_feed_quote_text`), **newest entry on top** under a `# Kern Errors` title — so a long API error survives the 3-second status bar (which is also blurred under the publish overlay's backdrop) and can be read/wikilinked in the editor like any note. `kern_error_log(context, detail)` in `textview.c` (declared in `editor_loop.h`); pure C, guarded on the documents dir being set (app always sets it; no-op in a bare CLI run), so it's headless-testable — publish-failure logging, newest-first ordering, feed-failure logging, and the no-docs-dir guard are covered in `tests/unit_textview.c`.
- **Publish confirmation overlay** — clicking "Publish" draws a modal **tweet preview** over the editor **in the main render loop** (not a macOS sheet): a blurred/darkened backdrop + an X-styled card showing the account's **avatar** (real `profile_image_url` photo, else an initials disc), **display name**, `@handle · <date>`, and the note body wrapped and **clipped to 400px**, with **Confirm** / **Cancel** buttons. State machine + geometry live in `textview.c` (`pub_state` = none/confirm/sending, `pub_text` snapshot; `pub_layout()` computes the card + button rects shared by the draw pass and the mouse hit-test; `draw_publish_overlay()` renders). Enter / click-Confirm posts via `kern_x_publish`; Esc / click-Cancel closes; the overlay is modal (top of the key/text/mouse routing). The async result arrives via `kern_x_publish_done(ok, info)` (Swift→C, any thread; stashed and applied on the main-thread tick in `pub_apply_result()`): on success it **copies the tweet URL to the clipboard** (`https://x.com/<handle>/status/<id>`), reports "Posted ✓", and closes; on failure it shows the error and returns the overlay to the confirm state to retry. Account identity comes from three Swift `@_cdecl` bridges — `kern_x_display_name()`, `kern_x_handle()`, `kern_x_avatar_rgba()` (fetched from `/2/users/me?user.fields=name,profile_image_url` at connect; the avatar is downloaded + decoded to straight-alpha RGBA with a **baked anti-aliased circular mask**, so the renderer draws it as a plain quad and the round photo is smooth). Adds one renderer-seam function, **`r_draw_image_circle`** (GL RGBA texture, cached by pointer; no-op headless stub). The state machine + confirm/cancel + result→clipboard→close + region-vs-whole + identity render are headless-tested in `tests/unit_textview.c` (platform-stub bridges return a fake identity + no avatar → the initials path).

## Commands

All test/perf/coverage commands run from `tests/` (that's where the Makefile lives).

```sh
# Headless test suite (ASan + UBSan; LSan on Linux). ~200 tests.
cd tests && make test

# Coverage report for the testable core (clang llvm-cov)
cd tests && make coverage
# Same report + enforce the >=85% line floor (what CI and the pre-push hook run)
cd tests && make coverage-check

# Performance harness (no sanitizers, -O2). Needs a corpus:
python3 tools/gen_prose.py 100MB test_100mb.txt   # from repo root; test_100mb.txt is gitignored
cd tests && make perf CORPUS=../test_100mb.txt    # add PERF_ARGS=--check for CI budgets

# Build the macOS app (fast, ~3s)
xcodebuild -project application/Kern/Kern.xcodeproj -scheme Kern -configuration Debug build CODE_SIGNING_ALLOWED=NO

# Release a Developer ID-signed Kern.app to ~/Desktop for local use (~15s)
scripts/release.sh            # bump minor version (.9 rolls to next major .0), build reset to 1
scripts/release.sh --no-bump  # rebuild the current version without bumping
```

- **Release pipeline** (`scripts/release.sh` + `scripts/ExportOptions.plist`): bumps the version, archives the Release config, exports a **Developer ID-signed** (not notarized) `Kern.app`, and copies it to `~/Desktop` (the user drags it into `/Applications` themselves). Version policy: minor +1 rolling at `.9`→next major `.0`, build reset to `1`; the bump is written to `project.pbxproj` **only on a successful build** (passed as `xcodebuild` overrides until then). No notarization — a locally-built app has no quarantine flag so it launches without a Gatekeeper prompt; to distribute to other Macs, add a `notarytool submit --wait` + `stapler staple` step.

- **Pre-push gate:** `.githooks/pre-push` runs all three CI checks locally before a push and aborts on failure. Activate once per clone with `git config core.hooksPath .githooks`; bypass a single push with `git push --no-verify`. It caches a 25MB perf corpus at `tests/perf_corpus.txt` (gitignored; override with `KERN_PERF_CORPUS`). **The hook and `ci.yml` run the *same* `make coverage-check` target so they can't drift** — the line-coverage floor is read from `llvm-cov export` JSON (`data[0].totals.lines.percent`, `tests/cov_floor.py`), which is stable across llvm versions. (This replaced a per-environment `awk '$10'` on the `llvm-cov report` table, whose column layout differs between macOS Apple-llvm and Linux llvm — it read *branch* coverage on CI and *line* coverage locally, so CI failed valid pushes the hook had passed.)
- **Running a single test:** the harness has no name filter. To narrow, comment out `RUN(...)` lines or `suite_*()` calls in `tests/test_main.c`.
- **Snapshot goldens:** after an *intentional* render change, regenerate with `KERN_UPDATE_SNAPSHOTS=1 make test` (goldens live in `tests/snapshots/`).
- **Run the built app:** `open` the product under `~/Library/Developer/Xcode/DerivedData/Kern-*/Build/Products/Debug/Kern.app`, or ⌘R in Xcode.

## How to verify a change (important)

The test build compiles the core **plus** `Editor/textview.c` — built with `-DKERN_HEADLESS_TEST`, which `#ifdef`s out only `editor_main`/`resize_event_watcher` (the real-SDL init + blocking loop) and exposes the `tv_test_*` seam. So:
- Changes to `buffer/editing/undo/navigation/md_render/recent/commands.c` → verify with `make test` (and `make coverage`).
- Changes to `textview.c`'s **event/dispatch + modal layer** (the `editor_handle_event` switch, prefix chords, wikilink nav, minibuffer, isearch, autosave in `editor_tick`) → covered by `tests/unit_textview.c`, which pumps synthetic `SDL_Event`s through `editor_handle_event` against the stub renderer + `tests/platform_stub.c`. Verify with `make test`.
- Changes to the parts still **not** compiled in tests — `editor_main` itself, the real-SDL render/init path, AppKit chrome, anything Swift — verify by building the app with `xcodebuild` and launching it. Confirm the app actually has a live pid before claiming it works.

## Architecture

**The renderer seam.** `Platform/renderer.h` is a 12-function interface (`r_draw_rect`, `r_draw_text`, `r_get_text_width`, `r_set_font_style`, `r_get_size`, …). The app implements it with SDL/GL (`Platform/renderer.c`); the tests implement it with a deterministic capture stub (`tests/stub_renderer.{c,h}`, 10×20px glyphs, 800×600 window, records every draw op). Any `Editor/` code that needs metrics or drawing goes through this seam, which is what makes layout testable headlessly.

**State lives in two structs.** `EditorState` (document model — buffer, cursor, mark, kill ring, undo; no SDL) and `ViewState` (UI — scroll, fonts, search/minibuffer/prefix flags, status). Defined in `Editor/editor_types.h`. **Every core function takes explicit `EditorState*`/`ViewState*` pointers.** The only globals are `g_ed`/`g_vs` singletons inside `textview.c`.

**Two tiers of `Editor/` code:**
- *Pure core* — `buffer.c` (lines, file I/O, kill buffer, mark/region, sandbox paths), `editing.c` (all `ed_*` ops), `undo.c` (operation-based, grouped). No renderer dependency.
- *Renderer-dependent but still headless-testable* — `navigation.c` (wrap, cursor/visual mapping, click-to-cursor, vertical movement, search), `md_render.c` (inline markdown spans + drawing), `commands.c` (the command table + `kern_dispatch_key`), `recent.c` (MRU). These depend only on `renderer.h` (plus `navigation.c` → `md_render.h`, since wrap/click/movement geometry must mirror the render — see the gotcha below), so they compile into *both* the app and the test binary. (`navigation.c`'s link graph now pulls `md_render.c` + `pos_render.c` + `style_check.c`; the perf harness in `tests/Makefile` links them too.)
- *Partly headless-testable shell* — `textview.c` (~1,400 lines): its per-event work (`editor_handle_event`) and per-frame work (`editor_tick`) are extracted from `editor_main` so the modal/dispatch layer (minibuffer, wikilinks, buffer-switch, prefix chords, isearch, autosave) is driven in `tests/unit_textview.c` via the `tv_test_*` seam (`Editor/editor_loop.h`). The remainder — `editor_main`'s real-SDL init + blocking loop and the GL render path — stays app-only.

**Command dispatch.** `commands.c` owns the single binding table `g_commands[]` + `kern_dispatch_key(ed, vs, kmod, sym)`. `textview.c`'s keydown handler calls `kern_dispatch_key` first; only things needing local UI state are handled outside it: `M-g` (minibuffer), the `C-x …` and `ESC`/meta prefix chords (`handle_*_prefix_key`), and the arrow keys (which delegate to the `commands.c` movement functions, layering shift-select + word-jump). Feature tests drive `kern_dispatch_key` directly in `tests/unit_commands.c`.

**Lazy per-line caches.** `Line` carries `wrap_count` (cached visual-row count, `-1` = dirty) and `md_spans`/`md_span_count` (cached inline-span map, `-1` = not computed). `line_dirty()` invalidates both on edit; `buf_invalidate_all_wraps()` clears wraps on a page-width or font-size change. `nav_maybe_reflow` skips the full re-wrap when a resize doesn't change page width.

**Determinism seams.** Time goes through `Editor/clock.h` (`kern_now_ms()`; app uses `clock_sdl.c`, tests use a settable `clock_fake.c`). Clipboard goes through `Editor/clipboard.h` (app `clipboard_sdl.c`, tests `clipboard_fake.c`). Use these, not SDL directly, in `Editor/` code.

## Conventions / gotchas

- **Always measure wrap and column math in `FONT_REGULAR`.** A frame ends with the status bar's `FONT_MONO` active; if an event-time wrap recompute measures in mono, it caches wrong wrap counts → phantom rows and mis-placed click-to-cursor. `navigation.c`'s `body_width()` and the wrap functions force `FONT_REGULAR` regardless of ambient style — preserve this in any wrap/metric change.
- **Navigation geometry must mirror what `do_render` draws.** Each visual row is drawn at `nav_page_margin() + md_row_indent(line, row_start)` (a list item's continuation rows hang under the item text) with **per-span font widths** (bold/italic/mono differ from `FONT_REGULAR`), and a heading's `### ` may hang in the left margin. Wrap-breaking, click-to-cursor, and the caret-x all go through that same origin + `md_x_to_col`/`md_col_x` (per-span aware) + `nav_heading_markers_hang` — otherwise clicks land offset on styled/list lines and wraps drift past the right edge. Keep these three in lockstep with the render pass; `md_row_indent` is the single source of the indent formula.
- **Vertical movement is by *visual* row, not logical line.** `cmd_next_line`/`cmd_previous_line` (and the up/down arrows, which route through them) call `nav_visual_move`, which steps one screen row and keeps a *pixel* goal column in `ViewState.goal_x/goal_line/goal_col` (reused only while the caret is still where the last vertical move left it). Moving a whole logical line on `C-n` inside a wrapped paragraph is the bug this replaced. Up past the top / down past the bottom land at buffer start/end.
- **Columns are byte-indexed**, not codepoint-indexed (UTF-8 inserts advance the cursor by byte length).
- **File paths are sandboxed** via `buf_set_documents_dir()` (set from Swift at launch); when unset (CLI/test builds) paths are used verbatim. User paths resolve under the documents dir via `buf_resolve_path`.
- **Test layout:** `tests/ed_fixture.h` provides `ed_load`/`ed_teardown`/`LINE()`; `tests/test.h` provides `CHECK`/`CHECK_IEQ`/`CHECK_SEQ` and `RUN`. Each `unit_*.c` exposes one `suite_*()` listed in `test_main.c`. Snapshots prove render parity; they re-implement the text pass, so a `do_render` text-loop refactor won't be caught by goldens.
- **Coverage reality:** core is ~99% lines / 100% functions; the residual uncovered lines/branches are malloc/IO-failure paths and defensive guards (need fault injection, not more tests). The gated coverage floor (≥85%, `make coverage`) measures `COVCORE` = the core only — `textview.c` is **deliberately excluded** so its partly-tested shell doesn't mask the core's saturation. `textview.c` is ~53% lines (event/dispatch + modal handlers tested; uncovered remainder is the `do_render`/`process_frame` GL chrome and the `#ifdef`'d-out `editor_main`); see it with `make coverage-textview`.
- **Pumping events in tests:** drive `editor_handle_event` with synthetic `SDL_Event`s (`tests/unit_textview.c` has `key()`/`textinput()`/`type()` builders). The modal-routing precedence is real — don't bypass it by calling `handle_*` directly. The wikilink dropdown (`wl_active`) is recomputed inside `do_render`, so activate it by typing then calling `editor_tick()` before sending the accept/dismiss key. `tv_test_reset()` clears `g_ed`/`g_vs` + all modal file-statics between tests; `platform_stub.c` fakes the SDL input-state calls and the `kern_x_*` bridge. **Keep `editor_main` loop-shape-preserving** (wait → drain via `editor_handle_event` → `editor_tick`) — that's what keeps it both snappy and testable.
- **The main thread is parked by `editor_main`.** It runs SDL's blocking event loop and never returns, so the main thread / main actor is stuck for the app's life. Any Swift feature added to the app **must run off the main actor and report UI through the C status bar** (`kern_x_set_status`). `DispatchQueue.main.async`, main-actor `Task {}`, SwiftUI live updates, and `ASWebAuthenticationSession` all silently deadlock. Background threads (`Task.detached`, GCD, `NWListener`, `URLSession`, `NSWorkspace.open`) work fine. The X publishing layer (`App/KernApp.swift`) is the reference pattern.
